TASK 4 — HuggingFace/keychain graceful fallback (Linux + Windows)
--------------------------------------------------------------------
Read crates/athena-app/src/keychain.rs fully first, as required. Backend (Rust) change
was necessary and unavoidable for this task specifically — no frontend code can catch
or recover from a native OS credential-store failure; only the Rust layer that calls
`keyring::Entry` can.

- MODIFIED crates/athena-app/src/keychain.rs (the only Rust file this task touches,
  besides the one flagged, unavoidable exception below)
    Verified against `keyring` 2.x's own published docs: its default ("v1") feature
    gives exactly the three native backends (macOS Keychain / Windows Credential
    Manager / Linux Secret Service) and ships no built-in file-based fallback option —
    so Task 4's option (a) does not apply, and option (b) (local encrypted-file
    fallback) was implemented instead, for every credential this module manages
    (GitHub token, the 3 OAuth token sets, both AI-provider keys) via three new
    internal helpers — `store_secret` / `load_secret` / `delete_secret` — that every
    existing accessor now routes through instead of calling `keyring::Entry` directly.

    Fallback activates only when `keyring::Error::PlatformFailure` or `::NoStorageAccess`
    is returned (verified these are the two actual "backend unavailable" variants in
    keyring's `Error` enum against its docs — not a `NotSupportedPlatform` variant,
    which does not exist) — never for `NoEntry` or any other error, so a genuine
    problem is never silently masked.

    Fallback store: AES-256-GCM (via the new `aes-gcm` dependency) encrypted JSON file
    at `<user-data-dir>/athena-app/keychain-fallback/fallback-secrets.json`, keyed by a
    32-byte key generated on first use and stored alongside it
    (`fallback.key`, owner-only 0600 permissions on Unix). Every fallback activation
    logs one `tracing::warn!("using local fallback storage, OS keychain unavailable")`
    per process (not per call) via the existing tracing setup.

    Windows: `keyring = "2"` with no feature flags already gets the Windows Credential
    Manager backend by default (verified via keyring's own crates.io/docs.rs
    description: the default feature works "on Mac, Windows, and *nix... using exactly
    the same API"); no Cargo.toml feature-flag change was needed for that. The same
    `store_secret`/`load_secret`/`delete_secret` fallback path applies unconditionally
    on Windows too, in case Credential Manager is ever unavailable there, per the
    task's explicit "add the same fallback there too" instruction.

- MODIFIED crates/athena-app/Cargo.toml (dependency only, as scoped)
    NEW DEPENDENCY: `aes-gcm = "0.10"` — a well-known, NCC-Group-audited RustCrypto AEAD
    crate, needed because `keyring` has no built-in file-fallback option (see above).
    Also added `rand = "0.8"`, `base64 = "0.22"` (both already present transitively in
    Cargo.lock via other dependencies, at compatible versions — added directly here
    only because keychain.rs now calls them directly) and `dirs = "5"` (resolves the
    OS-standard per-user data directory without needing an `AppHandle` threaded into
    keychain.rs, which takes none today).

- Frontend notice, per Task 4's explicit requirement ("Surface a clear, non-scary
  message in the frontend... if fallback storage is active"):
    FLAGGED DEVIATION FROM THE "keychain.rs + Cargo.toml only" SCOPE NOTE: satisfying
    this requirement is impossible without one new IPC command, since the frontend has
    no other way to learn fallback status. Rather than a new file, the single command
    was added as a one-line addition to crates/athena-app/src/commands/mod.rs (which
    already hosts one free-standing command, `get_app_version`, as precedent) plus its
    registration in main.rs's `generate_handler!` — the minimal possible exception,
    called out explicitly here rather than silently exceeding the stated scope.
        - MODIFIED crates/athena-app/src/commands/mod.rs: added
          `is_using_keychain_fallback` command, calling
          `keychain::is_using_fallback_storage()` (a new pub fn added in keychain.rs).
        - MODIFIED crates/athena-app/src/main.rs: registered it.
        - MODIFIED src/ipc/bindings.ts: added `isUsingKeychainFallback()`.
        - MODIFIED src/screens/Settings/index.tsx: checks it on mount; renders a plain-
          language notice ("Your device's secure keychain isn't available right now,
          so any keys or tokens you save below are being stored in an encrypted file
          in Athena's own app-data folder instead...") above the AI-providers section,
          where API keys are entered — satisfying the task's "where API keys are
          entered" placement.

Guessed/inferred values, flagged:
    - The exact AES-GCM file format (nonce || ciphertext, base64-encoded, one JSON map
      keyed by keychain username) is this implementation's own design — Task 4 left the
      encrypted-file format unspecified beyond "a minimal well-known crate."
    - `dirs = "5"` was chosen (not previously a dependency) as the minimal way to
      resolve a per-user data directory without threading `AppHandle` through
      keychain.rs; an alternative would have been to accept an `AppHandle` parameter
      into every keychain function, which would have required touching every call site
      (commands/ai.rs, commands/integrations.rs, oauth_loopback.rs) — a much larger,
      unscoped change Task 4 did not ask for.
